# Generative UI Prototype — Implementation Plan

## Concept

Build a sports app (ESPN-style) whose entire UI is generated by an LLM on each visit. The LLM takes in a user's click and search history and outputs a personalized React interface — hiding clutter, surfacing what matters, and inventing new components — all within a defined design system.

The core thesis: **start from everything (full ESPN), personalize by subtraction and reorganization.**

---

## Key Architecture Decisions

### 1. AI Output Format: Actual JSX Component Code
The AI returns real React component code, not a JSON layout spec. This is necessary because we want the AI to invent components on the fly (e.g., a calendar of upcoming men's basketball games) rather than pick from a pre-built menu.

**Risk:** Rendering dynamically generated JSX is dangerous with raw `eval()`.
**Solution:** Use a sandboxed renderer like [`react-live`](https://github.com/FormidableLabs/react-live) or render inside a sandboxed iframe. The AI should output self-contained components that only use injected props (data from the sports API) and design system classes/tokens.

### 2. Design System: Constraints, Not Components
Feed the AI a design system markdown file (colors, spacing scale, typography, border radius, motion tokens) so all generated components feel visually consistent — but place no limits on what components it can invent.

**What to include in the design system file:**
- Color palette (primary, secondary, neutral, accent, error)
- Typography scale (font family, sizes, weights, line heights)
- Spacing scale (4px base grid)
- Border radius tokens
- Shadow/elevation levels
- Breakpoints for responsive behavior
- Any animation/transition standards

The design system file gets injected into every LLM prompt as a system-level constraint.

### 3. Cold Start: Full ESPN Default
A brand-new user with zero history sees a full ESPN-equivalent layout — all sports, all menu items, general scores, top news, everything. This is the "maximum information" baseline state.

Every personalized UI is a subversion of this baseline: hiding irrelevant sections, foregrounding preferred content, and adding novel components the user's history suggests they'd want.

**Implementation:** The cold-start layout can either be:
- Hardcoded as a static default React page, or
- Generated by the LLM with an empty history prompt (simpler, more consistent with the overall architecture)

## System Flow

The data pipeline runs in two phases. The AI decides what to show and what data it needs; the app fetches exactly that data; the app fills the components with real values.

```
User opens app
       ↓
Load click/search history from Firebase (empty if new user)  ← RAG #1
       ↓
─── PHASE 1: Layout + Data Requirements ───
Prompt Gemini with: design system + clickstream history
Gemini returns JSON:
  {
    "jsx": "<SportsFeed component as string>",
    "espnEndpoints": ["sports/lacrosse/scoreboard", ...]
  }
       ↓
─── PHASE 2: Fetch + Render ───
App fetches exactly those ESPN endpoints                      ← RAG #2
Injects real JSON as props into the generated component
Renders in sandboxed JSX renderer (react-live)
       ↓
User interacts → clicks/searches logged to Firebase
       ↓
On next visit, repeat with updated history
```

**Why two phases?**
Pre-fetching all possible ESPN data on every load is wasteful and bloats the prompt. Instead Gemini acts as its own retrieval planner — it knows what it built and specifies exactly what live data is needed to fill it. This is **Agentic RAG**: the model decides what to retrieve rather than a fixed system doing it.

## Firebase User Context Storage

### Firestore Structure

```
users/
  {userId}/                        ← "user1" or "katie"
    profile/
      createdAt: timestamp
      preferencesSummary: string   ← AI-generated summary of older history
      summaryUpdatedAt: timestamp
    events/                        ← subcollection, one doc per interaction
      {eventId}/
        type: "CLICKED" | "SEARCHED" | "SCROLLED_PAST"
        label: string              ← human-readable (e.g. "Cornell vs Penn State")
        sport: string              ← e.g. "lacrosse", "nfl", null
        section: string            ← e.g. "hero", "scoreboard", "nav", "sidebar"
        timestamp: timestamp
    cachedUI/
      latest/
        jsx: string                ← last generated JSX (avoids regen on every load)
        generatedAt: timestamp
        eventCountAtGeneration: number
```

### Two Users

- **user1** — anonymous/temporary user. Starts fresh, accumulates context over time, can be fully reset via a button in the UI. Good for demonstrating the cold-start → personalized progression live.
- **katie** — pre-seeded with rich context reflecting real ESPN usage preferences (college lacrosse, golf, minimal NFL/NBA interest). Good for demonstrating how well the system works once it knows a user.

### How Context Is Built for the Prompt

**Layer 1 — Recent events (last 50, passed as-is)**
```
CLICKED: NCAA Men's Lacrosse — lacrosse — scoreboard — 2026-04-17T14:22Z
SEARCHED: Masters betting odds — golf — search — 2026-04-17T14:25Z
SCROLLED_PAST: NFL scores section — nfl — scoreboard — 2026-04-17T14:26Z
```

**Layer 2 — Preference summary (for older history)**
Once a user exceeds 50 events, older events are summarized by Gemini into a paragraph stored in `profile.preferencesSummary`:
```
"This user is primarily interested in college lacrosse (especially Cornell)
and golf. They consistently scroll past NFL and NBA content. They have
searched for betting odds twice. They prefer scoreboards over news articles."
```

### Caching

Only regenerate the UI when 5+ new events have accumulated since last generation. Otherwise serve the cached JSX with fresh ESPN data.

```
On page load:
  if (newEventCount >= 5 OR no cachedUI exists):
    → run full Phase 1 + Phase 2 pipeline, cache result
  else:
    → render cachedUI.jsx directly with fresh ESPN data
```

## Gemini Prompt Structure

### Phase 1 Prompt — Layout + Data Requirements

```
SYSTEM:
You are a UI engineer generating a personalized ESPN-style sports feed.
Respond with valid JSON only — no markdown, no explanation.
Response schema:
{
  "jsx": "<string — a complete React component as a string>",
  "espnEndpoints": ["<endpoint path>", ...]
}

Design system constraints:
[DESIGN_SYSTEM_TOKENS injected here]

JSX rules:
- The component must be named SportsFeed and accept one prop: { data }
- data matches the ESPN endpoint responses keyed by endpoint path
- Use only standard HTML elements and inline Tailwind classes
- Invent whatever components serve this user — do not limit yourself
- Default (zero history): render the full ESPN homepage equivalent

Available data shape (request what you need via espnEndpoints):
- scoreboard:  { events: [{ id, name, status, competitors: [{ team, score }] }] }
- news:        { articles: [{ headline, description, published }] }
- standings:   { entries: [{ team, wins, losses, rank }] }
- teamSchedule:{ events: [{ date, opponent, venue, result }] }

USER:
[PREFERENCE SUMMARY]
{preferencesSummary or "No history yet — show full ESPN default"}

[RECENT INTERACTIONS]
{last50Events or "No interactions yet"}

Generate a personalized sports feed. Surface what this user cares about,
hide what they ignore, invent components where useful.
```

## Sports Data Integration

### ESPN API (Free, No Key Required)

```
# Scoreboard
http://site.api.espn.com/apis/site/v2/sports/{sport}/{league}/scoreboard

# News
http://site.api.espn.com/apis/site/v2/sports/{sport}/{league}/news

# Standings
http://site.api.espn.com/apis/site/v2/sports/{sport}/{league}/standings

# Team schedule
http://site.api.espn.com/apis/site/v2/sports/{sport}/{league}/teams/{teamId}/schedule
```

Sport/league examples: `basketball/mens-college-basketball`, `football/nfl`, `baseball/mlb`, `hockey/nhl`, `golf/pga`

### How RAG Powers the Data Layer

**RAG #1 — Clickstream as User Knowledge Base**
Firebase history is retrieved at load time and injected into the Gemini prompt. No vector DB needed — 50 events fit directly in context. Same pattern as document RAG: retrieve → augment → generate.

**RAG #2 — ESPN API as Live Sports Knowledge Base (Agentic RAG)**
Gemini can't know today's scores from training data. Phase 1 asks Gemini what endpoints it needs; Phase 2 fetches them. The model decides what to retrieve — this is Agentic RAG, and a novel application of the pattern (UI generation vs Q&A).
